# Варианты вывода цен из БД на странице Diamant

## Исходные данные

Бот уже записывает каждое изменение цены в таблицу:

```text
diamant_gold_prices
```

Страница должна брать последнюю запись:

```sql
SELECT *
FROM diamant_gold_prices
ORDER BY id DESC
LIMIT 1;
```

Нужные поля:

```text
price_583_from / price_583_to
price_585_from / price_585_to
price_750_from / price_750_to
price_850_from / price_850_to
price_875_from / price_875_to
price_916_from / price_916_to
price_999_from / price_999_to
```

Для золотых коронок отдельной цены в таблице сейчас нет. Можно временно
использовать цену 850 пробы либо добавить отдельные поля.

## Вариант 1. Контроллер и модель OpenCart

Контроллер страницы вызывает модель, модель читает последнюю строку из
`diamant_gold_prices`, а Twig-шаблон выводит готовые значения.

Схема:

```text
MySQL -> OpenCart Model -> OpenCart Controller -> Twig -> HTML
```

Плюсы:

- цены находятся в HTML сразу при загрузке;
- работает без JavaScript;
- хорошо индексируется поисковиками;
- нет публичного API;
- можно добавить серверный кэш;
- наиболее правильная архитектура для OpenCart.

Минусы:

- нужно найти точный route и шаблон этой страницы;
- потребуется изменить файлы OpenCart;
- изменения темы могут потребовать переноса правки.

Это рекомендуемый вариант для постоянной работы.

## Вариант 2. Отдельный модуль OpenCart

Создаётся модуль `Gold Prices`, который читает таблицу и выводится в нужной
позиции страницы через макет OpenCart.

Плюсы:

- не нужно изменять ядро OpenCart;
- модуль можно включать и отключать из админки;
- удобно переносить между темами;
- можно сделать настройки и fallback-цены в админке.

Минусы:

- разработка дольше;
- текущую HTML-секцию цен придётся заменить выводом модуля;
- нужно корректно встроить существующую вёрстку.

Это лучший вариант, если цены позже будут использоваться на нескольких
страницах Diamant.

## Вариант 3. Read-only endpoint и JavaScript

На сайте создаётся endpoint:

```text
GET /api/get-gold-prices.php
```

Он возвращает последнюю цену в JSON. Страница выполняет `fetch()` и заменяет
статические значения после загрузки.

Схема:

```text
MySQL -> PHP endpoint -> JSON -> JavaScript -> HTML
```

Плюсы:

- почти не требует вмешательства в OpenCart;
- можно вставить в существующий HTML страницы;
- быстро реализуется;
- удобно проверять отдельно.

Минусы:

- сначала пользователь может увидеть старые fallback-цены;
- при отключённом JavaScript останутся старые цены;
- появляется дополнительный HTTP-запрос;
- endpoint нужно защищать от лишней нагрузки;
- поисковый робот может увидеть fallback вместо актуальной цены.

Подходит как быстрый временный вариант.

## Вариант 4. JSON-файл без запроса к MySQL со страницы

После успешной синхронизации PHP endpoint записывает цены и одновременно
создаёт статический файл:

```text
/api/current-gold-prices.json
```

Страница читает этот файл через JavaScript.

Плюсы:

- страница не открывает соединение с MySQL;
- файл быстро отдаётся nginx/Cloudflare;
- минимальная нагрузка;
- даже при временной недоступности MySQL последняя цена остаётся доступна.

Минусы:

- всё ещё нужен JavaScript;
- нужно атомарно обновлять файл;
- БД и JSON могут разойтись при ошибке записи файла.

Это хороший компромисс для большого трафика.

## Вариант 5. Серверный include-компонент

HTML страницы переносится в PHP/Twig-шаблон, который подключает небольшой
компонент цен. Компонент сам читает последнюю строку БД.

Плюсы:

- актуальные цены сразу в HTML;
- код цен изолирован в одном файле;
- проще варианта с полноценным модулем.

Минусы:

- PHP не выполняется внутри обычного описания информационной страницы;
- нужно менять файл шаблона темы;
- прямой запрос из представления хуже разделения Model/Controller/View.

Допустимо как промежуточное серверное решение.

## Вариант 6. Обновлять HTML страницы после изменения курса

После изменения цены скрипт находит информационную страницу OpenCart в БД и
заменяет в её HTML старые цифры новыми.

Плюсы:

- публичная страница остаётся полностью статической;
- не требуется запрос к таблице цен при каждом открытии;
- не нужен JavaScript.

Минусы:

- хрупкая замена HTML;
- легко повредить вёрстку или ручные изменения;
- необходимо очищать кэш OpenCart/темы;
- история и отображаемый HTML могут разойтись;
- трудно поддерживать несколько языков.

Не рекомендуется для постоянной эксплуатации.

## Вариант 7. Хранить актуальную цену в настройках OpenCart

Endpoint сохраняет историю в `diamant_gold_prices` и дополнительно обновляет
текущие значения через настройки OpenCart. Шаблон читает настройки.

Плюсы:

- история остаётся в отдельной таблице;
- текущая цена доступна стандартными средствами OpenCart;
- удобно использовать на нескольких страницах.

Минусы:

- появляются две копии актуальных данных;
- нужно следить за синхронностью;
- потребуется очищать кэш настроек.

Подходит, если цены нужны во многих существующих модулях OpenCart.

## Вариант 8. Кэшировать серверный HTML-фрагмент

OpenCart-модель читает цену из БД, формирует HTML блока и кэширует его на
несколько минут. При синхронизации кэш принудительно очищается.

Плюсы:

- правильный серверный HTML;
- низкая нагрузка на MySQL;
- актуальность контролируется синхронизацией.

Минусы:

- сложнее обычного Model/Controller;
- нужно учитывать разные языки и темы;
- ошибка очистки кэша временно оставит старую цену.

Подходит при большом трафике после внедрения варианта 1 или 2.

## Что делать не стоит

- Не вставлять данные подключения MySQL в JavaScript.
- Не подключать браузер напрямую к MySQL.
- Не выполнять PHP внутри HTML-описания из админки: OpenCart его не исполняет.
- Не читать БД REG.ru напрямую из браузера.
- Не давать публичному endpoint возможность записывать цены.
- Не использовать таблицу истории без `ORDER BY id DESC LIMIT 1`.
- Не удалять статические fallback-цены до проверки рабочего решения.

## Рекомендация

После проверки локальной копии установлено, что страница Diamant выводится не
из `information_description`, а модулем Journal3 Blocks `Landing Gold`
(`module_id=352`). Поэтому для этой страницы выбран серверный OCMOD над
`catalog/controller/journal3/blocks.php`.

План реализации:

1. Установить подготовленный OCMOD для Journal3 Blocks.
2. В модуле `Landing Gold` заменить числа метками `{{gold_price_...}}`.
3. При рендеринге блока читать последнюю строку `diamant_gold_prices`.
4. Заменять метки сервером до отправки HTML пользователю.
5. Использовать статические fallback-значения при пустой таблице.
6. Очистить кэш Journal3 после установки и изменения HTML.
7. Проверить русскую и узбекскую версии страницы.
8. Не менять логику бота и запись истории цен.

Готовый пакет и инструкция находятся в
`site_integrations/diamant/price_page_ocmod/`.
